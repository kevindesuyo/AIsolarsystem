# ステークホルダー合意メモ

## 合意事項サマリ
- 全体ロードマップは5フェーズ構成 (0〜4) とし、各フェーズ完了時にレビュー + レトロを実施する。
- 教育・デモ用途に耐える安定性とアクセシビリティを最優先とし、性能指標とUX指標を並行で追う。
- ツールチェーン整備 (Phase1) が後続フェーズの前提条件であり、完了まで大規模な機能開発を凍結する。
- Worker化および物理エンジン刷新 (Phase2) はPoCで効果を確認しつつ段階導入。現行実装はロールバック手段として保守。
- 国際化・アクセシビリティ (Phase3) は教育現場のフィードバックを元に優先度を再評価し、テスト観点をQAと連携する。
- 品質保証 (Phase4) ではGitHub Actionsによる自動テスト/ビルド/配布フローと監視体制を整備し、ローンチ条件を満たすまでリリースしない。

## 優先順位
1. 基盤整備 (型、Lint、CI)
2. 物理エンジンの安定化と性能指標改善
3. UX/アクセシビリティ改善
4. 配布・監視体制

## レビュー/意思決定体制
- プロダクトオーナー (Kevin): フェーズごとのゴール承認、優先度調整
- 技術リード (Codex): 技術アーキテクチャ決定、リスク承認、設計レビュー
- 実装担当: プルリク作成、自己レビュー、テスト証跡の提示
- QA: Phase4で主導、E2Eテストおよび回帰検証の承認

## コミュニケーションルール
- 週次ステータスレポート (金曜EOD) で進捗・リスク・ブロッカーを共有
- Issueテンプレートとラベル体系を整備し、チケット駆動開発を徹底
- 大きな設計変更はADRまたはDocsフォルダに記録し、関係者が非同期で確認できるようにする
- 緊急対応や方向転換が必要な場合はSlack #solarsystem チャンネルで即報告し、24時間以内に方針決定

## 未解決事項
- QAリソース確保のタイミング (Phase3後半〜Phase4初頭で調整)
- 教材化/公開範囲のライセンス方針
- モニタリング基盤 (Sentry / LogRocket / 独自メトリクス) の予算・運用担当
